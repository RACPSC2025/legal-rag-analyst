## 1. Visión Estratégica: "The Golden Markdown Library"

Para un sistema de análisis legal profesional, el costo y la precisión son los dos pilares fundamentales. Leer PDFs directamente en cada consulta es ineficiente y costoso (en tokens y tiempo de procesamiento). 

### 1.1 El Concepto
Implementar una **Capa de Persistencia Intermedia en Markdown (.md)**. 
- **Entrada:** PDF original (sucio, con tablas complejas, múltiples columnas).
- **Proceso:** Conversión de alta fidelidad con **Docling** (IBM).
- **Salida:** Archivo `.md` perfectamente estructurado, legible y "congelado".

### 1.2 Ventajas Competitivas
1.  **Reducción Drástica de Costos:** Leer texto plano (.md) consume hasta un **60-80% menos de tokens** que procesar un PDF con técnicas de visión o OCR en tiempo real.
2.  **Fidelidad Estructural:** Docling preserva la jerarquía (Capítulos, Artículos) y las tablas legales, que son el "talón de Aquiles" de los PDFs.
3.  **Auditoría y Edición:** Los archivos `.md` permiten una revisión humana rápida. Si la conversión tuvo un error en un número de artículo, se corrige en el Markdown y la base de datos vectorial se actualiza instantáneamente.
4.  **Velocidad de Respuesta:** El sistema no tiene que "re-descubrir" la estructura del documento en cada consulta; ya está pre-digerida.

## 4. Análisis Senior: Ronny Camacho — Especificaciones de Implementación

### 4.1 Núcleo de la Golden Library (`src/services/markdown_service.py`)

El servicio se divide en tres componentes críticos:

1.  **`MarkdownLibrary`**: Gestiona la biblioteca en disco utilizando un `_manifest.json`. Registra cada archivo mediante un **hash SHA-256** (16 caracteres). El esquema de nombres es `{stem}_{hash8}.md` para garantizar la unicidad absoluta. Cada archivo `.md` incluye un encabezado estilo YAML con metadata (`source_pdf`, `processed_at`, `source_tool`, `file_hash`), la cual se elimina automáticamente durante la lectura para consultas.

2.  **`MarkdownConversionService`**: Implementa `convert_and_save()`, que detecta si ya existe una "Golden Copy" basándose en el hash del archivo (independientemente del nombre) para evitar re-procesamientos innecesarios. 
    - **Motor principal:** Docling con `do_table_structure=True` y `do_cell_matching=True`.
    - **Fallback:** Uso de `pdfplumber` para detectar tablas con `page.extract_tables()` exportándolas como Markdown (`| col | col |`), además de detectar patrones de `ARTÍCULO/CAPÍTULO` para convertirlos en encabezados `##`.

3.  **`query_markdown_direct()`**: Reemplazo inteligente de la consulta directa a PDF. Identifica automáticamente las "Golden Copies" de los archivos PDF y utiliza un `_MARKDOWN_QUERY_PROMPT` especializado para preservar la integridad de las tablas. Reporta un `token_source: "markdown"` para feedback visual en la UI.

### 4.2 Evolución del Pipeline (`src/ingestion/ingest_pipeline.py v2`)

El pipeline ahora integra `_resolve_paths()`, separando las fuentes en archivos `.md` directos y PDFs pendientes de conversión. 
- Los archivos `.md` se procesan con `MarkdownHeaderTextSplitter` (respetando la jerarquía `#` a `####`), seguido de un `RecursiveCharacterTextSplitter` aplicado únicamente a fragmentos que excedan el `chunk_size`. 
- Los PDFs sin copia Markdown siguen el flujo de carga tradicional.

### 4.3 Interfaz de Usuario Avanzada (`app.py v2`)

Se introduce la pestaña **🟡 Biblioteca Golden MD**, con las siguientes funcionalidades:
- Conversión individual o por lotes.
- Ingesta automática a ChromaDB post-conversión.
- Tabla de gestión de archivos con opciones de descarga, vista previa y eliminación.
- Sidebar con toggle **"Preferir Golden MD"**, incluyendo un indicador visual del ahorro de tokens obtenido en cada respuesta.

**1. MarkdownLibrary (El Guardián de la Biblioteca):**
Utiliza un sistema de **Hashing SHA-256** truncado a 16 caracteres. Esto es fundamental para la integridad legal: un archivo se identifica por su ADN binario, no por su nombre en disco. El `_manifest.json` permite una persistencia tipo base de datos para recuperar metadata (fecha, herramienta, tamaño) sin tocar los archivos `.md`.

**2. MarkdownConversionService (La Refinería de Formato):**
Implementa una orquestación entre **Docling 2.0** (motor de IBM) y un fallback en **pdfplumber**. 
*   **Docling:** Extrae el layout semántico (encabezados, listas) y utiliza `TableFormer` para garantizar que las tablas de resoluciones no pierdan su estructura.
*   **Fallback pdfplumber:** Si el entorno carece de recursos, este módulo detecta por patrones (`re.compile`) palabras clave como "ARTÍCULO" o "RESUELVE" para reconstruir la jerarquía Markdown manualmente.

**3. query_markdown_direct (El Motor de Consulta Económica):**
Permite saltarse la base de datos vectorial para consultas puntuales. 
*   **Ahorro de Tokens:** Al procesar Markdown plano en lugar de PDF, el ahorro estimado es del **70%**.
*   **Fidelidad:** El prompt obliga al LLM a devolver el texto **literal** y las tablas **íntegras**, algo vital para un analista legal que necesita la cita exacta.

## 8. Ingeniería de Datos: Principios Generales para Chunking Legal (2026)

### 8.1 Principios Fundamentales
| Principio | Recomendación | Razón Jurídica | Impacto en RAG |
| :--- | :--- | :--- | :--- |
| **Estructura Lógica** | No cortar cláusulas, artículos o el "RESUELVE". | Preserva la jerarquía normativa. | Evita perder contexto de obligaciones. |
| **Contexto Cruzado** | Overlap + Metadata rica (Artículos, NITs, Fechas). | Referencias multi-hop entre normas. | Mejora precisión en preguntas complejas. |
| **Tamaño Óptimo** | 400-800 tokens (1500-3000 chars) con 20% overlap. | Equilibrio semántico y técnico. | Evita truncation en Nova/Titan. |
| **Integridad de Tablas** | Mantener tablas completas (usar Parent-Child). | Datos técnicos densos en resoluciones. | Tablas rotas = pérdida grave de info. |
| **Enriquecimiento** | Metadata: Sección, Resolución, Entidad, Art/Cláusula. | Vital para filtrado dinámico. | Esencial para CRAG y citas. |

### 8.2 Estrategias Recomendadas (Orden de Prioridad)

1.  **Hierarchical / Structure-Aware (Prioridad 1):** Basada en la Golden Markdown Library. Segmenta por encabezados (# RESOLUCIÓN, ## CONSIDERANDO, ### ARTÍCULO) respetando la semántica legal nativa.
2.  **Parent-Child Retrieval (Modo Pro):** Child chunks para búsqueda precisa + Parent chunks para entrega al LLM. La mejor solución para tablas y artículos extensos.
3.  **Semantic Chunking + Metadata:** Uso de similitud semántica para detectar cambios de tema, blindado con reglas legales para no cortar párrafos críticos.
4.  **Recursive Fallback:** Uso de `RecursiveCharacterTextSplitter` con separadores legales colombianos: `["\n## RESUELVE", "\n## CONSIDERANDO", "\n### ARTÍCULO", "\nTabla "]`.

## 9. Mejores Prácticas Adicionales y Estrategia Híbrida "Fénix"

### 9.1 Mejores Prácticas de Ingeniería
*   **Parent-Child + Context Enrichment:** Búsqueda en micro-chunks para precisión, entrega de secciones completas (Parent) al LLM. Implementar ventana de contexto dinámica (±2 chunks).
*   **Metadata de Grado Legal:** Inyectar campos específicos en cada chunk: `resolution_number`, `entity` (SDA/CAR), `date`, `nit`, `volume_m3`, `section`.
*   **Post-Chunk Validation:** Verificación con LLM ligero (Temp 0) para asegurar que la segmentación no rompa obligaciones o tablas.
*   **Métricas RAGAS:** Priorizar **Faithfulness > 0.90**. Medir *Context Recall* antes y después de cada cambio en el chunker.
*   **Overlap Estratégico:** 20-25% (aprox. 180 tokens) para garantizar la continuidad en referencias cruzadas.

### 9.2 Estrategia Híbrida Ganadora (Veredicto Final)
1.  **Docling** como motor de conversión a Markdown estructurado.
2.  **AdaptiveChunker** con separadores legales + Recursive splitter como salvaguarda.
3.  **Parent-Child Retrieval** para equilibrio entre búsqueda precisa y contexto rico.
4.  **Enriquecimiento de Metadata** masivo por cada fragmento.
5.  **Context Enrichment** (ventana ±2) para máxima cohesión.
